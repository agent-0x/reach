# Reach v0.2.0 — AI-Native Tools Design Spec

## Goal

Add 6 new MCP tools that give AI agents capabilities SSH structurally cannot provide: structured system observability, command safety analysis, transactional execution with rollback, and multi-server orchestration.

## Guiding Principle

**SSH returns text for humans. Reach returns structured JSON for AI.**

Every new endpoint returns machine-parseable JSON. AI agents should never need to parse human-readable text output.

---

## Feature 1: `reach_logs` — Structured Log Query

### Why
AI agents currently run `journalctl` via `reach_bash` and parse messy text. This tool returns structured JSON log entries with filtering and cursor-based incremental reads.

### API

**Agent endpoint:** `POST /logs`

```json
// Request
{
  "service": "reach-agent",     // optional, omit for system-wide
  "lines": 50,                  // default 50, max 500
  "since": "5m",                // relative: "5m","1h","2d" or ISO8601
  "until": "",                  // optional, ISO8601
  "grep": "error",              // optional, case-insensitive substring
  "cursor": ""                  // optional, from previous response
}

// Response
{
  "entries": [
    {
      "timestamp": "2026-03-30T14:22:01.123Z",
      "service": "reach-agent",
      "priority": "err",
      "message": "AUTH_FAIL from 1.2.3.4: invalid token on /exec"
    }
  ],
  "count": 1,
  "cursor": "s=abc123;i=456",
  "has_more": false
}
```

### MCP Tool

```
reach_logs(server, service?, lines?, since?, until?, grep?, cursor?)
```

### Platform Backends

| Platform | Implementation |
|----------|---------------|
| Linux | `journalctl --output=json` with flag mapping |
| macOS | `log show --style=ndjson` with flag mapping |
| Windows | `wevtutil qe` XML → JSON conversion |

### Architecture

```
internal/agent/
├── logs.go             — handler + LogBackend interface
├── logs_linux.go       — journalctl implementation
├── logs_darwin.go      — macOS log implementation
└── logs_windows.go     — wevtutil implementation
```

**Interface:**
```go
type LogEntry struct {
    Timestamp string `json:"timestamp"`
    Service   string `json:"service"`
    Priority  string `json:"priority"`
    Message   string `json:"message"`
}

type LogResult struct {
    Entries []LogEntry `json:"entries"`
    Count   int        `json:"count"`
    Cursor  string     `json:"cursor"`
    HasMore bool       `json:"has_more"`
}

type LogBackend interface {
    Query(service string, lines int, since, until, grep, cursor string) (*LogResult, error)
}
```

### Cursor Semantics (P1-6 fix)

Cursors are **best-effort**, not strongly consistent. The contract:

- **Linux**: Uses native journald cursor (stable, exact, guaranteed unique).
- **macOS/Windows**: Synthetic cursor = `(timestamp_ns, platform_record_id)`. On macOS, `log show` assigns monotonic IDs per boot. On Windows, `wevtutil` entries have `EventRecordID`. These provide a stable tiebreaker beyond timestamp. Resume uses `>` on the record ID within the same timestamp, guaranteeing forward progress without skipping siblings.
- **If platform record ID unavailable** (edge case): Falls back to `(timestamp_ns, hash(message)[:8])`. Documented as best-effort — rare skips possible, but forward progress guaranteed.
- **Deduplication**: Guaranteed on Linux (native cursor). Best-effort on macOS/Windows — no silent skips, rare duplicates acceptable.
- **Ordering**: Entries are always returned in chronological order (oldest first).
- **Cursor invalidation**: If the cursor points to a rotated/purged log region, the response includes `"cursor_valid": false` and `"oldest_available": "2026-03-30T00:00:00Z"`. The entries field is **empty** — the client must explicitly re-query with a `since` parameter. **No silent fallback.** This prevents the client from thinking it has continuous coverage when it doesn't.
- **Conflicting parameters**: If `cursor` is provided alongside `since`/`until`, the cursor takes precedence for the start position; `until` still applies as the end bound. `since` is ignored when `cursor` is present.

## Feature 2: `reach_services` — Service Status Aggregation

### Why
AI agents run `systemctl status X` per service, parse multiline text, repeat N times. This returns all services at once with structured metrics.

### API

**Agent endpoint:** `POST /services`

```json
// Request
{
  "services": ["reach-agent", "nginx", "fail2ban"],  // optional, empty = all
  "filter": "all"               // "all" | "active" | "failed" | "inactive"
}

// Response
{
  "services": [
    {
      "name": "reach-agent",
      "status": "active",
      "sub_status": "running",
      "pid": 1234,
      "memory_mb": 12.5,
      "cpu_percent": 0.3,
      "uptime_secs": 86400,
      "restarts": 0,
      "enabled": true
    }
  ],
  "summary": {
    "total": 3,
    "active": 2,
    "failed": 1,
    "inactive": 0
  }
}
```

### MCP Tool

```
reach_services(server, services[]?, filter?)
```

### Platform Backends

| Platform | Implementation |
|----------|---------------|
| Linux | `systemctl show` + `/proc/<pid>/stat` for CPU/mem |
| macOS | `launchctl list` + `launchctl print` |
| Windows | `sc query` + WMI `Win32_Service` |

### Architecture

Existing `reach_info` returns basic info. AI agents need CPU usage %, memory usage %, top processes, network/disk IO — all as numbers they can directly compare and alert on.

### MCP Tool

```
reach_stats(server, top_n?)
```

### Security: Process Command Lines (P1-7 fix)

`top_processes` returns only `pid`, `name`, `cpu_percent`, `memory_mb`. **No command line / argv.** Process command lines frequently contain credentials, tokens, and connection strings. Exposing them in a routine monitoring endpoint is an unnecessary security risk. If AI needs to know what a process is doing, it can use `reach_bash` to run `ps` with explicit intent.

### Implementation
All via gopsutil — already cross-platform (Linux/macOS/Windows). No build tags needed.

```
internal/agent/
└── stats.go            — handler, uses gopsutil for everything
```

## Feature 4: `reach_dryrun` — Command Pre-Check / Risk Scoring

### Why
SSH fires-and-forgets. AI agents should be able to ask "is this dangerous?" before executing. This extends the existing blacklist into a risk analysis system.

### API

**Agent endpoint:** `POST /dryrun`

```json
// Request
{
  "command": "rm -rf /opt/old"
}

// Response
{
  "risk": "high",                // "low" | "medium" | "high" | "blocked"
  "score": 85,                   // 0-100
  "reasons": [
    "Recursive delete (rm -rf)",
    "Targets a top-level directory"
  ],
  "affected": {
    "type": "directory",
    "path": "/opt/old",
    "files": 847,
    "size_mb": 234
  },
  "suggestion": "Consider listing contents first: ls -la /opt/old",
  "would_execute": true          // false if blocked by blacklist
}
```

### Risk Scoring Rules

| Pattern | Risk | Score |
|---------|------|-------|
| Blocked by blacklist | blocked | 100 |
| `rm -rf` on non-system dir | high | 80-90 |
| `rm` (non-recursive) | medium | 40-50 |
| `systemctl restart/stop` | medium | 50-60 |
| `chmod`/`chown` changes | medium | 40-50 |
| `apt/yum install/remove` | medium | 50 |
| `kill`/`pkill` | medium | 50-60 |
| Config file writes (`/etc/*`) | medium | 40-50 |
| Read-only commands (`ls`, `cat`, `ps`) | low | 0-10 |
| Unknown commands | medium | 30 |

For file-affecting commands (`rm`, `mv`, `cp`), the agent stats the target path to populate `affected` with file count and size.

### Shell Parsing & Bypass Prevention (P1-3 fix)

The risk scorer operates on **two levels**:

1. **Blacklist check**: Uses existing `CheckDangerous()` regex patterns on the raw string. This catches `rm -rf /`, `mkfs`, etc. regardless of wrapper.
2. **Heuristic scoring**: Splits the command on shell operators (`&&`, `||`, `;`, `|`) and scores each simple command independently. The highest score wins.

**Known limitations (documented, not hidden):**
- `sh -c '...'`, `bash -c '...'`: The scorer extracts and re-scores the inner string.
- Variable expansion (`$VAR`), globbing (`*`), command substitution (`` `...` ``): **Not resolved.** Commands containing these are scored as `medium` (score 50) minimum with a reason: `"Contains unresolved shell expansion — risk may be higher than scored"`.
- Symlink resolution: The `affected` path analysis resolves symlinks via `filepath.EvalSymlinks` before stat.

The response includes a `confidence` field:
```json
{
  "confidence": "high"    // "high" | "medium" | "low"
}
```
- `high`: Command is a simple command, no expansion, path is concrete
- `medium`: Contains pipes/chains but each part is parseable
- `low`: Contains variable expansion, subshells, or complex syntax — treat score as lower bound

**This is a heuristic safety signal, not a sandbox.** It helps AI agents make better decisions, but does not replace the blacklist for enforcement.

### MCP Tool

```
reach_dryrun(server, command)
```

### Architecture

```
internal/agent/
├── dryrun.go           — handler + risk scoring engine + shell splitter
└── dryrun_test.go      — test risk patterns including bypass attempts
```

Reuses `exec.go`'s `CheckDangerous()` for blocked detection. New risk scoring on top.

## Feature 5: `reach_transaction` — Sequential Command Execution with Best-Effort Rollback

### Why
SSH executes commands one by one. If step 3 of 5 fails, you're left in a dirty state. Reach can execute a sequence with automatic rollback on failure.

**Important:** This is NOT database-style ACID. It provides sequential execution + compensating rollback. The term "transaction" is used in the sense of "a unit of work", not "serializable isolation". The API documentation and MCP tool description must reflect this.

### API

**Agent endpoint:** `POST /transaction`

```json
// Request — nginx config update with validation
{
  "steps": [
    {
      "command": "cp /etc/nginx/nginx.conf /tmp/nginx.conf.pre-txn",
      "rollback": "cp /tmp/nginx.conf.pre-txn /etc/nginx/nginx.conf"
    },
    {
      "write": { "path": "/etc/nginx/nginx.conf", "content": "..." }
      // No rollback needed — agent auto-backs up before write
    },
    {
      "command": "nginx -t"
      // No rollback needed — read-only validation
    },
    {
      "command": "systemctl reload nginx"
    }
  ],
  "timeout": 60
}

// Response — success
{
  "status": "completed",
  "steps": [
    {"index": 0, "command": "cp ...", "exit_code": 0, "stdout": ""},
    {"index": 1, "write": "/etc/nginx/nginx.conf", "exit_code": 0},
    {"index": 2, "command": "nginx -t", "exit_code": 0, "stdout": "syntax ok"},
    {"index": 3, "command": "systemctl reload nginx", "exit_code": 0, "stdout": ""}
  ],
  "rollback_results": [],
  "status": "completed"
}

// Response — validation failed, auto-rollback
{
  "status": "rolled_back",
  "failed_step": 2,
  "error": "nginx: configuration file test failed",
  "steps": [
    {"index": 0, "command": "cp ...", "exit_code": 0},
    {"index": 1, "write": "/etc/nginx/nginx.conf", "exit_code": 0},
    {"index": 2, "command": "nginx -t", "exit_code": 1, "stderr": "nginx: configuration file test failed"}
  ],
  "rollback_results": [
    {"index": 1, "action": "restore from auto-backup", "exit_code": 0},
    {"index": 0, "command": "cp /tmp/nginx.conf.pre-txn /etc/nginx/nginx.conf", "exit_code": 0}
  ]
}

// Response — rollback itself partially failed
{
  "status": "partial_rollback",
  "failed_step": 2,
  "error": "nginx: configuration file test failed",
  "steps": [...],
  "rollback_results": [
    {"index": 1, "action": "restore from auto-backup", "exit_code": 0},
    {"index": 0, "command": "cp /tmp/nginx.conf.pre-txn ...", "exit_code": 1, "stderr": "No such file"}
  ]
}
```

### Terminal States (P0-1 fix)

```
completed        — all steps succeeded
rolled_back      — failed, all rollbacks succeeded
partial_rollback — failed, some rollbacks also failed (DIRTY STATE)
rollback_failed  — failed, rollback could not even start (e.g. timeout)
```

The response always includes `rollback_results` with per-step exit codes so the caller knows exactly which rollbacks succeeded and which failed. **The agent never lies about state** — if rollback partially failed, the status says `partial_rollback`, not `rolled_back`.

### Rollback Contract

- Rollback only runs for **successfully completed steps** (not the failed step).
- Rollback runs in **reverse order** of successful completion (last completed → first).
- Steps without a `rollback` field are skipped during rollback.
- **Rollback commands must be idempotent.** The spec example is updated to demonstrate this: instead of multiple steps sharing the same `mv` command, each step has a distinct rollback or the backup uses a unique path.
- If a step has `write` field, the agent auto-generates a backup before writing (saved as `<path>.reach-bak.<txn-id>`) and the auto-rollback restores from this backup. No user-supplied rollback needed for write steps.

### Isolation (P1-1 fix)

**Best-effort, single-writer serialization per agent:**
- The agent holds a transaction mutex. Only one `/transaction` can execute at a time per agent instance.
- Concurrent `/transaction` requests receive `423 Locked` with `{"error": "transaction in progress", "retry_after_ms": 1000}`.
- **`/exec` and `/write` are NOT blocked** during a transaction. However, the auto-backup system uses a **compare-before-restore** check: before restoring a backup, the rollback verifies the file's current SHA256 matches the version that the transaction wrote. If it doesn't match (i.e., something else modified the file after the transaction step), the rollback **skips that file** and reports it in `rollback_results` with `"skipped": true, "reason": "file modified externally"`. This prevents rollback from clobbering concurrent changes while being transparent about what it couldn't restore.
- The MCP tool description explicitly states: *"Best-effort rollback. Concurrent modifications to the same files may prevent full rollback."*

### Timeout & Process Lifecycle (P1-2 fix)

- **Per-step timeout**: Each step inherits the remaining time from the total `timeout`. If a step exceeds the remaining time, its process group is killed (same as `/exec` behavior) and the step is treated as failed.
- **Rollback timeout**: Rollback gets a **separate 30-second deadline** (not deducted from the original timeout). Each rollback step gets min(30s / remaining_rollback_steps, 15s).
- **Process kill before rollback**: The agent waits for confirmed process exit (process group SIGKILL + waitpid) before starting rollback. No rollback begins while a step's process is still alive.
- **Rollback step failure**: If a rollback step fails or times out, it is recorded in `rollback_results` and the next rollback step still executes. After all rollback attempts, the status is `partial_rollback` if any failed.

### Execution Logic

1. Acquire transaction mutex (or return 423)
2. Execute steps sequentially, each through blacklist check
3. For `write` steps: auto-backup target file before writing
4. If any step returns non-zero → stop forward execution
5. Kill step process group, wait for exit confirmation
6. Execute rollback for **successfully completed steps** in reverse order (30s deadline)
7. Release transaction mutex
8. Return status with full step + rollback results

### MCP Tool

```
reach_transaction(server, steps[], timeout?)
```

### Architecture

```
internal/agent/
├── transaction.go      — handler + sequential executor + rollback + mutex
└── transaction_test.go — test rollback logic, partial failure, timeout
```

## Feature 6: `reach_orchestrate` — Multi-Server Orchestration

### Why
AI agents managing multiple servers need to coordinate deployments. Currently they call `reach_bash` in a loop. Orchestrate provides rolling/parallel strategies with health checks.

### API

This is a **client-side / MCP-side** feature. No new agent endpoint needed — it calls existing agent endpoints.

**MCP Tool:**

```
reach_orchestrate(servers[], command, strategy, health_check?, timeout?)
```

```json
// MCP Request args
{
  "servers": ["radar", "sniper"],
  "command": "systemctl restart btc_pm",
  "strategy": "rolling",        // "rolling" | "parallel"
  "health_check": "curl -sf localhost:8003/health",  // optional
  "health_timeout": 30,         // seconds to wait for health check
  "timeout": 60                 // per-server command timeout
}

// Response
{
  "status": "completed",        // "completed" | "partial_failure" | "aborted"
  "results": [
    {
      "server": "radar",
      "command_result": {"stdout": "", "exit_code": 0},
      "health": "ok",
      "duration_ms": 3200
    },
    {
      "server": "sniper",
      "command_result": {"stdout": "", "exit_code": 0},
      "health": "ok",
      "duration_ms": 2800
    }
  ]
}
```

### Strategies

- **rolling**: Execute on one server at a time. Wait for health check to pass before proceeding to next. On failure: stop and report (default) or continue if `continue_on_error: true`.
- **parallel**: Execute on all servers simultaneously. Collect results. Report all regardless of individual failures.

### Health Check Model (P1-5 fix)

Health check is always a **shell command executed via `client.Exec()`** on the target server. The `health_check` parameter is a command string (e.g., `curl -sf localhost:8003/health`). This is simpler and more flexible than using the agent's `/health` endpoint — the caller decides what "healthy" means.

- Health check runs after the command succeeds (exit_code 0).
- Health check is retried every 2 seconds up to `health_timeout` (default 30s).
- Health check passes when exit_code == 0. Fails when timeout expires.
- If no `health_check` is specified, the command result alone determines success.

### Failure Policy

- **rolling + failure**: Stop at the failed server. Already-completed servers are NOT rolled back (no implicit undo). The response status is `partial_failure` with full results showing which servers succeeded and which failed. The AI agent decides what to do next.
- **rolling + `continue_on_error: true`**: Skip failed server, continue to next. Status is `partial_failure` if any server failed, `completed` if all succeeded.
- **parallel + failure**: All servers execute regardless. Status reflects aggregate outcome.

**No implicit rollback across servers.** Multi-server rollback is a separate deliberate action the AI agent should take using `reach_transaction` per server if needed. Automatic fleet-wide rollback is too dangerous to be implicit.

### Architecture

```
internal/mcp/
└── orchestrate.go      — MCP tool handler, calls client.Exec per server, manages rolling/parallel
```

No agent changes needed. Pure client-side orchestration using existing agent APIs.

## Cross-Cutting Concerns

### Version & Capability Discovery (P1-4 fix)

Agent version bumps to `0.2.0`. The `/health` endpoint is extended:

```json
{
  "ok": true,
  "version": "0.2.0",
  "capabilities": ["logs", "services", "stats", "dryrun", "transaction"]
}
```

**Client behavior:**
- On first connection (or when cached capabilities are stale), the client calls `/health` and caches `capabilities`.
- MCP tools check cached capabilities before calling the agent. If a capability is missing (old agent), the tool returns a clear error: `"server 'X' does not support reach_logs (agent v0.1.0, needs v0.2.0)"`.
- `reach_orchestrate` checks capabilities on ALL target servers before starting. If any server lacks the needed capability, it reports the incompatibility upfront instead of failing mid-rollout.

### Agent Server Registration
Register 5 new routes in `server.go`: `/logs`, `/services`, `/stats`, `/dryrun`, `/transaction`.

### MCP Registration
Register 6 new tools in `mcp/server.go`: `reach_logs`, `reach_services`, `reach_stats`, `reach_dryrun`, `reach_transaction`, `reach_orchestrate`.

### Client Methods
Add 5 new methods to `client.go`: `Logs()`, `Services()`, `Stats()`, `DryRun()`, `Transaction()`. Orchestrate uses existing `Exec()` + `Health()`.

### Platform Build Tags
Files with `_linux.go`, `_darwin.go`, `_windows.go` suffixes use Go's filename-based build constraint convention. No explicit `//go:build` tags needed.

### Backward Compatibility
- All existing endpoints unchanged
- `reach_info` kept as-is; `reach_stats` is its superset
- New agent version serves both old and new endpoints

### Testing Strategy
- Unit tests for risk scoring patterns (dryrun)
- Unit tests for transaction rollback logic (mock executor)
- Unit tests for log entry parsing per platform
- Integration test: full agent serve + client call cycle
